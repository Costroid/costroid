# Costroid — Architecture & Build Spec (canonical)

The canonical **technical** source of truth for the build. This lives in the repo as `ARCHITECTURE.md` and governs every technical decision below it. It supersedes the earlier strategy docs for build purposes; launch/marketing material stays separate. **Scope and sequencing now defer to [PRODUCT-PLAN.md](PRODUCT-PLAN.md)** — the going-forward, step-by-step production plan (the taskbar, the connections subsystem, the generalized quota model, and the two deferred providers all live there); this doc stays the technical authority and is trued up to that plan. Code-shaped notes describe intent and naming, not frozen signatures. Detailed companion specs (DATA-MODEL.md, DESIGN-SYSTEM.md) are referenced where they own the granular truth.

---

## 1. What Costroid is (and is not)

**Position:** the local, private FinOps tool for AI coding assistants — the one that *also* sees the subscription limits the platforms can't.

**It is NOT:** a platform, observability, enterprise FinOps, a proxy/gateway, a chatbot or conversational-LLM interface, or anything that processes prompt/completion content. **Metadata only.** (Any web platform is a separate, separately-licensed repo — never built here.)

**Competitive frame:** measured against **ccusage** and the status-bar tools — not Finout, OpenCost, or Vantage. The two load-bearing differentiators, because no billing-based competitor can do them: **subscription-quota awareness** (no invoice exists for it) and **local-first / no content / zero integration**.

**Audience:** individual developers, for now.

## 2. The intellectual core: the cost–quality frontier

Price and quality don't track each other, and that gap is why this product exists.

- DeepSWE (avg cost/task, updated 2026-05-30): gpt-5.5 = 70% @ $6.61 vs claude-opus-4.7 = 54% @ $18.19 — gpt-5.5 cheaper *and* better; claude-sonnet-4.6 = 32% @ $5.52 (cheapest → on-frontier). DeepSWE's cost column is cache-miss-priced (~5×, indicative). opus-4.8 isn't benchmarked here yet → show as a gap, never a guessed number (§9.3).
- CursorBench v3.1 (2026-05-18): Composer 2.5 = 63.2% @ $0.55 vs Opus 4.7 Max = 64.8% @ $11.02 (matched at ~1/20th the cost); gpt-5.5 = 64.3% @ $4.37. Vendor benchmark (Cursor's own private tasks, non-reproducible, showcases Composer) — corroborating only; its **cost is cache-aware** (computed from published per-token pricing), so unlike DeepSWE the cost is sound — the caveat here is provenance, not cost.

The frontier is jagged: the cheapest model is sometimes a trap, the priciest often isn't best. So the product **shows the frontier, plots where the user's spend sits on it, and lets them judge** — never "use the cheapest." It takes model *quality* from the benchmarks but the *dollar* axis from Costroid's own cache-correct cost (a benchmark task-average isn't the user's actual bill; §12 has the per-benchmark cost caveats). It stays honest that it sees spend and benchmarks but *not* task difficulty, so it informs rather than prescribes. That honesty is what makes it trustable, and this principle governs the whole UI — not just one screen.

## 3. Scope (v1)

*Current state: this v1 core is built and **shipped through v0.3.0** (with the T7/T8 connect groundwork since landed on `main`); the live "where are we" is [PRODUCT-PLAN.md](PRODUCT-PLAN.md) §0. The list below is the v1 scope as designed.*

- **Providers:** Claude Code + Codex (validated), Cursor (v1: **detection only** — present + selected model, labeled beta; its usage/quota are served live server-side with no sanctioned source, so they stay **"unavailable"** — discovery-gated, §8, never via session reuse).
- **Cost:** subscription quota (%) *and* API cost ($); filter by provider. Cross-provider totals are **per-lane** (see §6 — API spend, subscription-equivalent value, and quota % shown distinctly), never one merged number.
- **Trends dimensions:** aggregate by period (`day` / `week` / `month` / `year`) and group (`model` / `app` / `total`).
- **Export:** FOCUS records to JSON and CSV.
- **Surfaces:** the TUI (`now` / `trends`, with the recommendation/frontier surface reached via the `a` ask key + the insight line); a `costroid statusline` mode (tmux, Starship, Claude Code's `statusLine`); and **`costroid setup-statusline`** — a first-class onboarding command that wires Claude Code's `statusLine` (the capture snippet into an existing one, or Costroid as the statusline) so live Claude quota works. A cross-platform **taskbar/tray app is planned** — the last surface (PRODUCT-PLAN.md Step 6, target v0.6.0), built in **egui / eframe + the `tray-icon` crate** (all MIT/Apache-2.0; no Tauri, no webview) as package `apps/bar` (binary `costroid-bar`); it consumes the core every other surface already shows.
- **Recommendation:** a frontier view from **DeepSWE (primary)** + **CursorBench (corroborating)**, as a bundled snapshot.

## 4. Technical canon (constraints the build inherits)

- Rust; single Cargo workspace; edition 2021; **MSRV 1.88** (documented in CI); workspace crates versioned **in lockstep** (`[workspace.package].version`); commit `Cargo.lock`. **CI gate (green = releasable):** `rustfmt` + `clippy -D warnings` + `cargo test` + FOCUS-conformance + `cargo deny` (license + advisories) + offline-acceptance (no-network).
- `thiserror` in libs (`ProviderError`/`FocusError`/`CoreError`), `anyhow` in the binary. **No `unwrap`/`expect`/`panic` in libs** (tests use the `match … panic!` idiom).
- `tracing` with an env filter, **local diagnostics only, never networked**; quiet by default, `-v`/`-vv` to stderr (**planned convention — not wired yet:** nothing links `tracing` today and the CLI defines no verbosity flags; adopt this shape when diagnostics are first needed). **No telemetry.** License **Apache-2.0**; dependencies must be **permissively licensed** (MIT / Apache-2.0 / BSD / ISC / Zlib / Unicode) — **no copyleft** (GPL / AGPL / LGPL / SSPL); verify before adding (the CI gate runs `cargo deny`).
- Standards: FinOps Foundation + **FOCUS 1.3** (validator `finopsfoundation/focus_validator`, run offline against the **vendored 1.3.0.1 ruleset** at `scripts/focus-ruleset/` — the PyPI wheel bundles only 1.2.0.1, so a `--block-download` 1.3 run needs the vendored model; 3 known validator-ruleset defects allowlisted with upstream refs — only the rules that fire, never their passing siblings). Treat the FOCUS spec, not the docs, as the authority on column semantics.
- Pricing **bundled inside `costroid-core` via `include_str!`** at `crates/costroid-core/pricing/pricing.vN.json` — **cargo packages only files under the crate dir, so all bundled data must live inside its crate** (a workspace-root location breaks `cargo publish`). Sourced current at build time and refreshed deliberately — never hardcoded figures that drift; works fully offline. Records `as_of` + `sources`; rates per meter (input/output/cache_read/cache_write) **per 1M tokens**; the calculator derives the **per-token** unit price (rate ÷ 1e6). Catalog key = model id (e.g. `claude-sonnet-4-6`), not display name. Cost is always labeled an **estimate**; reconciliation against the real invoice is Phase 2+.
- **Config (planned convention — not wired yet):** TOML at the XDG path (`~/.config/costroid/config.toml`) with sensible zero-config defaults; no config system is built today — everything runs zero-config on built-in consts. **Secrets never go in config — keychain only.**
- crates.io: **iterate the existing names** (`costroid`, `costroid-core`, `costroid-focus`, `costroid-providers`) — published through 0.3.0; the next iteration is 0.4.0, which extends the publish ladder with `costroid-connect` (T10b, RELEASING.md) — don't orphan or rename them. npm: `costroid`.
- **Brand:** monochrome (black/grey/white, adapting to the terminal theme) + a **single amber accent reserved for the warning/near-limit state** (red only as an intensification, always with a non-color cue); JetBrains Mono; braille glyphs (U+2800 + bitmask); **mandatory `--plain` fallback** (linear labeled text, no color, no braille — §7). Mark `C⠉` (a pixel `C` beside the braille cell `⠉`); wordmark `costroid` with `cost` in the strong weight and `roid` muted — carry the split into the UI: **dollar figures and the active metric use the strong weight; labels and context are muted.** (Dot math + component specs in DESIGN-SYSTEM.)
- **Log discovery (WSL-aware):** Claude Code = `~/.claude/projects/**/*.jsonl` + `~/.config/claude/projects/**/*.jsonl`; Codex = `~/.codex/sessions/**/*.jsonl`; Cursor = **no local usage/quota** — Cursor serves these live from its own backend (not to local logs), and Costroid does **not** call Cursor's internal endpoints, so Phase 1 only **detects presence + selected model** from the `~/.cursor` config (honor `CURSOR_DATA_DIR`), never chat content. **Honor `CLAUDE_CONFIG_DIR` (comma-separated roots) and `CODEX_HOME` before the defaults; merge all roots.** Detect WSL via `/proc/sys/kernel/osrelease` (contains `microsoft`/`WSL`); under WSL, **scan `/mnt/c/Users/*` for the profile(s) that actually hold the logs** (so a WSL user ≠ Windows profile, e.g. `eren`/`ereno`, still resolves), with `USERPROFILE` (if set) and the legacy `$USER` path as fallbacks. Native roots are XDG / `~/Library` / `%APPDATA%`. Never assume a single fixed path. Claude Code keeps **no quota in its session logs**, but its live 5h/7d limits arrive *locally* through its **`statusLine` hook** — the `rate_limits` block (`five_hour`/`seven_day` → `used_percentage` 0–100 + `resets_at` epoch s; Pro/Max, after the first API response; **zero API tokens**), captured by the statusline shim (§7/§8), so **no credential-reuse endpoint is needed for Claude**. Codex exposes its rate-limit windows in the session logs directly (5h = 300 min, weekly = 10080 min).

## 5. Workspace architecture

- **`costroid-core`** — cost calc + FOCUS normalization orchestration + bundled pricing + a **`bench`/`recommend` module** (the DeepSWE/CursorBench snapshot, the bench overlay-reconciliation rule, the frontier computation), the T9b **`vendor_report`** types (the provider-neutral invoice side), and the T9c **`reconcile`** module (the estimate-vs-invoice reconciliation **engine** — pure-core, fixture-tested, consumes `vendor_report`; it added **no** dependency, so the `connect → core` direction is unchanged). No terminal/UI code. *Ported and test-guarded.* The trust foundation. Promote `bench` to its own crate only if it grows.
- **`costroid-focus`** — the FOCUS record types and their (de)serialization (JSON/CSV). Pure data, **no internal dependencies**; the crate most likely extracted/reused later. *Ported.*
- **`costroid-config`** — the shared, read-only user-config crate (the `[budget]`/`[alerts]` TOML schema + loader). **Landed in T20** (Step 6), extracted verbatim from `apps/cli/src/config.rs` so **both apps** (`apps/cli` + `apps/bar`) consume one schema with no drift and the taskbar needs no `apps/cli` dependency. Pure local-only leaf: depends on `costroid-core` (for the config-neutral `BudgetTargets`/`AlertThresholds` input types) + `serde`/`toml`/`rust_decimal`; **no network/keychain code**, no writer (still read-only — a missing file is the zero-config default, a malformed one a typed `ConfigError`). Direction `config → core`, never the reverse.
- **`costroid-providers`** — the provider trait + one adapter per provider + WSL-aware discovery. No internal dependencies (it emits the provider-neutral `UsageEvent`/`LimitWindow`; FOCUS normalization lives in `costroid-core` — the once-declared `costroid-focus` dep was unused and removed in the 2026-06-10 fix pass). Claude + Codex *ported/validated*; Cursor *new (beta)*. Live quota enters via **sanctioned/local sources only** — the Claude `statusLine` push and Codex's local windows (§4/§8); Cursor has no such source, so its live quota is **"unavailable"** (discovery-gated — pursued only if Cursor ships a sanctioned API; never via session reuse, §8). Costroid never reuses credentials against undocumented endpoints.
- The binary crate is **`costroid`** (the `cargo install costroid` / `npx costroid` name) — clap CLI + the Ratatui TUI + the statusline + `--live` + the `--plain` path. It lives at **`apps/cli`** (package `costroid`, binary `costroid`).
- `fixtures/` holds committed sample provider logs for tests — **never real user data**.
- **`costroid-connect`** — all network + credential code, feature-gated and **off by default**. The crate is feature-gated (skeleton landed in T7); its **keychain behavior landed in T8** — `keyring` 3.6.3 (sync Secret Service on Linux via C libdbus, Apple/Windows native backends) providing `CredentialStore` + a non-secret `ConnectionRegistry` + the `ApiVendor` enum — and its **network half's foundation landed in T9a**: the generic authorized-host HTTPS client (`AuthorizedClient` + `AuthHeader`/`HttpResponse`/`RequestLimits`) on blocking `ureq` + `rustls` (no async runtime), OS-native trust roots via `rustls-native-certs` (never `webpki-roots`), HTTPS-only + GET-only, redirects and env-proxies disabled, off-host requests refused in the type **before any I/O**, secrets carried as `secrecy::SecretString` header values that never reach logs/`Debug`/error text. **T9b added the per-provider usage-API adapters** — `AnthropicAdapter` + `OpenAiAdapter` (and the Gemini first-class-"unavailable" state) — which fetch a stored admin key's usage/cost reports and parse them into the **`costroid-core::vendor_report`** provider-neutral shapes; this is where `costroid-connect` gains its **first internal dependency, `costroid-core`** (direction `connect → core`, never `core → connect`; `costroid-core` is already in every build's resolved graph, so the offline.rs / cargo-deny graphs are unchanged). **T10a/T10c then wired the adapters' first callers** — the `apps/cli` `connect`/`disconnect`/`connections` CLI (T10a, with Anthropic's connect-time `validate()` → `OrgValidation`) and the `reconcile` display (T10c, via the injectable `AdapterSet::cost_report` seam) — so a network call now occurs **only** on an explicit, user-initiated `connect` / `connections --check` / `reconcile` action (behind `--features connect`); the default build and every other command still make zero. The crate names neither `chrono` nor `rust_decimal` directly (all date/money mechanics live in `costroid-core`, the lean-deps choice; `costroid-focus` remains unneeded). All at Step 4 (v0.4.0). The sync keyring backend and blocking ureq pull **no async runtime**, so `async-io`/`tokio` stay forbidden even under `--features connect`. The `connect` feature is gated on the **apps** (`apps/cli` and, since T20, `apps/bar`), off by default — a virtual workspace root holds no `[features]`. **`apps/bar`** (binary `costroid-bar`, egui/eframe + `tray-icon`) is now **in the workspace** — the taskbar surface at Step 6 (v0.6.0), built across T18 (scaffold) → T19 (Overview) → T20 (the shared `costroid-config` crate + the `active_alerts` banner + the four live panels). It depends on `costroid-core` + `costroid-config` (and, under its own `connect` feature, `costroid-connect` for the **display-only, zero-network** Providers connection lane — local keychain/registry reads only). No `costroid-mcp` in any scope (the crates.io name stays intentionally unclaimed; the recommendation engine it would have exposed is built into the `bench` module).

**Dependency direction (no cycles):** `costroid-focus` and `costroid-providers` sit at the bottom and depend on nothing internal; `costroid-core → providers + focus`; **`costroid-config → core`** (the read-only TOML config leaf, T20); the apps → `core` + `config` (`apps/cli` also → `providers`); `apps/bar → {core, config}`; nothing depends on a binary. When the `connect` feature is on, an app also → `costroid-connect` → `costroid-core` (the app gates it; `costroid-connect` publishes after `costroid-core`, per RELEASING.md) — still acyclic. **T9b made this edge live:** the usage-API adapters parse into `costroid-core::vendor_report`, so `costroid-connect` now depends on `costroid-core` (and only it — not `costroid-focus`, which no shape needs). T8's keychain credential store and T9a's generic HTTP client had deliberately added no internal dependency (a generic client needs no internal type); the adapters are the first code that does.

**Data flow (the default / local-only build makes no network calls — still enforced by the strace offline-acceptance test + the forbidden-crates test):** `discover()` → `parse` usage + limits → normalize to FOCUS + estimate cost → aggregate (period, group) → render (now/trends/statusline) **or** export (JSON/CSV). Renderers and exporters are pure consumers of the same in-memory model. **Network happens only through `costroid-connect`** — the crate landed its off-by-default skeleton in T7, its keychain credential store in T8, its generic authorized-host HTTP client in T9a, the per-provider usage-API adapters in T9b, and their first callers in T10a/T10c (the `connect`/`disconnect`/`connections` CLI and the `reconcile` display); a network call now occurs **only** on an explicit, user-initiated `connect` / `connections --check` / `reconcile` action to a provider endpoint the user authorized, behind the Cargo feature. The static offline test (`apps/cli/tests/offline.rs`) was **already re-scoped in T7** into a two-tier resolved-graph check — the default build forbids all networking/TLS/telemetry (incl. the `ureq`/`rustls`/`keyring` trio) and asserts `costroid-connect` is not linked; `--features connect` is checked as a **subset-allowlist** (`CONNECT_ALLOWED` — the reviewed connect-delta over the default graph, asserting the sanctioned trio is linked while async runtimes/OpenSSL/other HTTP clients/telemetry stay forbidden) — and the dynamic strace acceptance script grew a feature-on baseline in T8 (the `--features connect` build leaks no network and writes no `$HOME` residue on a normal run — since T9a that build links the HTTP client, so the baseline also proves the client existing ≠ a call happening), and T10a replaced the connect-action STUB with a network-namespace fail-closed check (`connect` with a fake key cannot reach the host and leaves no `$HOME` residue; `disconnect` makes no network call). No telemetry, ever.

**Provider trait semantics:** `discover()` returns "no data" (not an error) when a provider isn't installed — missing providers are skipped, not fatal. `costroid-core` holds a registry of providers; the CLI runs all detected or a subset via flags. Adding a provider should require **no changes outside `costroid-providers`** (the test of the abstraction). Limits come from local data in Phase 1 — including Claude's live 5h/7d via the sanctioned `statusLine` push (§8) and Codex's local windows; a provider with no sanctioned source (Cursor) stays detect-only, its quota "unavailable" (§8) — Costroid never reuses credentials to fetch it. **Three providers ship today (Claude Code, Codex, Cursor).** **GitHub Copilot** and the **Antigravity CLI** are *planned* — added via the **`Capability` descriptor** — now a required `capability()` method on the `Provider` trait, landed in T3 (PRODUCT-PLAN.md §2b, §8) — but **only after a live-install discovery confirms each one's real data/auth/quota shape; never build either adapter speculatively.**

## 6. Data model (summary — full spec in DATA-MODEL.md)

- **FOCUS output.** Each unit of API usage is emitted as FOCUS 1.3 Cost & Usage rows with `ChargeCategory = "Usage"`. Input / output / cache-read / cache-write each become a **separate row** (distinct `SkuId`/`SkuPriceId` + `x_TokenType`) so quantities and unit prices stay coherent. Use the active 1.3 participating-entity columns **`ServiceProviderName` / `HostProviderName` / `InvoiceIssuerName`**; the deprecated `ProviderName` / `PublisherName` are **also emitted** (mirroring the active values) only because the bundled 1.3 validator requires their presence — drop them with a 1.4 ruleset. `ServiceCategory = "AI and Machine Learning"`. Unit prices are **per token**; `PricingUnit = "tokens"`.
- **Custom `x_` columns:** `x_Model`, `x_TokenType` (input|output|cache_read|cache_write), `x_AccessPath` (api|subscription|unknown), `x_Estimated`, `x_PricingStatus` (priced|missing_price|unknown_model), `x_Tool` (claude-code|codex|cursor), `x_Project`, and **`x_ConsumedTokens`** — the raw token count, **always populated** (the aggregation engine totals from this).
- **Subscription limits are a separate type, not FOCUS rows.** A `LimitWindow` carries `tool`, optional `plan`, `kind` (`FiveHour` | `Weekly` | `Daily` | `Monthly` | `BillingCycle`), `measure: Option<LimitMeasure>` (a `TokenFraction` (0–1) **or** `Spend { used_usd, included_usd }` — a request-count measure is the legacy pre-June-2026 Copilot model, intentionally **not** implemented), `resets_at`, plus `captured_at: DateTime<Utc>` + `status: LimitStatus { Verified, Unverified, Unavailable }` — **no dollars**, never summed into a bill. **This generalized shape landed in T2** (PRODUCT-PLAN.md §2a / §3 T2), replacing the old `used_fraction`. The core `LimitAvailability` separately gains `Unverified` + `Estimated` variants — **at the availability/render layer only, never on the provider `LimitWindow`**. Their **live producers landed in T4** (the cross-check demote `Verified→Unverified`, the stale age-out, and the volume-based `Estimated` fallback); the **rendering of these states landed in T6** (the now-screen + statusline render all five arms — `Unverified` carries a `? unverified` cue + neutral meter, `Estimated` shows volume + `~$value` with no meter, `Spend` shows `$used / $included used`).
- **Three lanes, never summed across each other:** (1) **API usage** → FOCUS rows, real cost estimate, `x_AccessPath = "api"`; (2) **subscription usage** → FOCUS rows valued at **API-equivalent** (`x_Estimated = true`, `x_AccessPath = "subscription"`) — labeled estimated value, **never a bill**; (3) **subscription quota** → `LimitWindow`s (%). The cross-provider "total" is therefore **per-lane** (API spend, subscription-equivalent value, quota % shown distinctly). Recommendations attach **only** to `x_AccessPath = "api"` rows.
- **Access path is detected from evidence, never guessed.** Codex: `rate_limits` windows present ⇒ subscription. Claude Code: from auth mode via non-secret **presence** signals only (never read credential values). `unknown` when there's no signal.
- **Unpriced rows** (no matching rate, or empty pricing table): the four cost columns stay present and `0`, `SkuPriceId` is null, and the FOCUS-required pricing/consumption columns go null accordingly — but `x_ConsumedTokens` still carries the count, so unpriced usage is **never dropped** from totals; flag `x_PricingStatus`. **Never substitute a guessed price.**
- **Export contract:** JSON is a wrapper `{ "focusVersion": "1.3", "rows": [...] }` (never a bare array); CSV uses the exact FOCUS PascalCase header (`x_` columns appended). `LimitWindow`s export **separately**, never mixed into the cost data.
- **Grouping:** by `x_Model`, by `x_Project` (bucket `"unknown"` when undeterminable), or `total`; period buckets by `ChargePeriodStart` in the user's local time zone. Never sum `LimitWindow` data.

DATA-MODEL.md remains authoritative for the column-by-column mapping, the `UsageEvent`/`FocusRecord`/`LimitWindow` Rust shapes, per-provider field paths, the pricing JSON schema, and the FOCUS-validator nullability/defect mechanics.

## 7. Rendering & UX (mechanics here; dot math + mockups in DESIGN-SYSTEM)

- **Ratatui + crossterm.** Braille is computed directly from `U+2800` + an 8-dot bitmask — **not** Ratatui's braille constants or `Canvas` (those drift across versions, and `Canvas` is TUI-only). The **sparkline is hand-rasterized to braille** like every other component, so the one-shot and TUI paths stay identical (see the styled-document rule below).
- **Fill = bright vs dim cells.** A terminal cell has a single foreground color, so braille gives sub-cell *shape* but not sub-cell *color*: used vs remaining is **bright vs dim cells** at cell granularity (the solid-dot/hollow-ring look from the mockups is raster-only — website/marketing). **Under `NO_COLOR`, only ANSI styling is dropped — braille meters remain** and stay readable because used vs remaining are *glyph-distinct* (full `⣿` vs the `⣀` track), never color-alone; the ASCII `[####--]` rendering is reserved for braille-incapable terminals (`RenderMode::Ascii` — the fully-ASCII *visual* fallback, same layout with ASCII meters).
- **Warning thresholds:** `warn = 0.80`, `critical = 0.95` (defaults, configurable). The warning state turns amber (red at critical/over) **and always carries a textual cue** — `!`, `!!`, `OVER`, or "near limit" — so it survives `NO_COLOR`, color-blindness, and `--plain`. Cost bars never go amber (amber is for limits, not spend).
- **`RenderMode`:** `Braille` (default) / `Ascii` (the fully-ASCII visual fallback when braille is unsupported — internal, not a user flag) / `Plain` (`--plain`: no TUI chrome, no color, no braille, no meters — linear labeled text, screen-reader friendly; Costroid-generated text in it is pure ASCII, pinned by test, while provider-supplied names pass through verbatim). Mode is chosen from `--plain`, TTY detection (non-tty ⇒ Plain), `NO_COLOR`, and a braille-capability check. The only user-facing mode flag is `--plain`.
- **Interactive vs one-shot:** on a TTY, `now`/`trends` run as a navigable TUI; piped output, `--plain`, `statusline`, and `export` render once and exit.
- **One styled document, two adapters:** the renderer emits a neutral styled document (semantic styles: strong/dim/warn/critical/plain). A one-shot adapter serializes it to an ANSI/plain string; the TUI adapter maps it to Ratatui — keeping both identical. The one-shot serializer is snapshot-tested as the compatibility contract.
- **Terminal is always restored** (raw mode off, alternate screen left, cursor shown) on quit, error, and panic — via a restore guard plus a panic hook that leaves the alternate screen, chains the default handler, and exits 101. Works over SSH and inside tmux.
- **`--live`** re-collects data periodically (~2s); without it the screen is a snapshot refreshed manually (`r`).
- **`costroid statusline`** prints a single compact line (shell prompts, tmux, Starship) from the same core data and honors `RenderMode`. It is fast, and side-effect-free **on interactive stdin** (tmux/Starship — stdin is never read); with **piped** stdin (Claude Code's `statusLine` JSON) it opportunistically captures the `rate_limits` block into the local no-secret cache before rendering (path 2, T5 — that capture is the point of the surface). It also supports `--capture-only` (write the cache and exit) and `--wrap '<cmd>'` (capture, then delegate to an existing status-line command); all three paths degrade to a blank line + exit 0 on failure, never breaking the user's prompt. A configurable format string / presets is a planned addition, not yet built.
- **Screens.** `now` = stacked 5-hour + weekly **limit meters with reset countdowns**, then **API spend by model** (cost bars), then one insight line; limits and costs are visually parallel but clearly separate (limits carry no dollars). `trends` = a period **sparkline**, then the **breakdown cost bars**, then one insight line.
- **Keybindings:** `d`/`w`/`m`/`y` set period; `g` cycles group; `tab` switches screen; `f` or `/` filters (fuzzy model/app); `a` asks (hands context to the recommendation view, §10 step 4); `r` refreshes; `q`/`Ctrl-C` quits (always restoring the terminal); `?` help.
- **States:** *loading* = braille spinner + short label; *empty* = plain-language note on what Costroid looked for and how to point it at the data (incl. the WSL/Windows path note), never an error; *partial* = show what's available and **label the gap explicitly, never fabricate** (e.g. Cursor); *per-provider error* = inline, non-fatal, the rest still renders.

## 8. Security boundary & credentials

The whole trust story depends on this.

- **Secrets live only in the OS keychain** (`keyring` crate: macOS Keychain / Windows Credential Manager / Linux Secret Service). **Never written to disk, config, or logs.**
- **Credentials flow only between the device and the provider** — there is no Costroid backend or server.
- **Sanctioned channels first — Costroid does not reuse a provider's credentials when a provider-built or local source exists.** Both validated providers are covered *inside the Phase-1 trust envelope* (no Costroid network call, no credential touched): **Claude Code's `statusLine` hook** — an extension point Anthropic builds and documents for third-party scripts — pushes a live `rate_limits` block (`five_hour`/`seven_day`, each `used_percentage` 0–100 + `resets_at` epoch s; Pro/Max only, after the first API response; **runs locally, zero API tokens**); and **Codex's session logs** already carry the 5h/weekly windows (§4). Costroid captures the Claude numbers without a subprocess where possible: the lower-friction path is a **one-line capture snippet the user adds to their existing statusline** (it `tee`s the `rate_limits` block into a no-secret local cache — Anthropic's own documented caching pattern); for users without one, **Costroid *is* the statusline** (renders + captures natively). **The cache** is a JSON file at `${XDG_STATE_HOME:-$HOME/.local/state}/costroid/claude-rate-limits.json` (Linux-side only — it is Costroid's own state) holding a top-level `captured_at` (RFC3339) plus `five_hour`/`seven_day` windows, each `used_percentage` (0–100) + `resets_at` (epoch seconds **or** RFC3339); it holds **no secret**. **T4 reads + sanitizes it** (absent/unreadable → Unavailable, never an error); **T5 shipped the writer** — `costroid setup-statusline` wires Claude Code's `statusLine` to capture into this cache (via `costroid statusline --capture-only`, or by Costroid *being* the status line), so the cache populates after the next Pro/Max response; **T6 renders it on the now-screen** (the Available/Unverified/Estimated arms with the always-on "as of HH:MM" freshness stamp and the claude.ai chat caveat sub-note). *Wrapping* another command is a hazardous fallback only (you must tee stdin or the wrapped command renders blank, and you stack latency inside Claude Code's status-line debounce budget). Never an Anthropic endpoint call with Claude Code's token — the **opposite of the codexbar credential-reuse pattern**, and the canonical Claude-quota mechanism. **Coverage decision (Opus weekly).** The hook carries `five_hour`/`seven_day` only — **no per-model weekly window** (Opus or Sonnet). So **lead with the overall 7d % as the real number Costroid can measure — not as "your binding cap"**: for a ~97%-Opus user the **Opus weekly may bind first**, and that sub-cap's % is exactly what the hook can't see. **The Opus-specific render is design intent, not yet shipped** (STATUSLINE-CAPTURE-BRIEF §6's `opus_weekly` field): the design shows Opus 7d **volume + estimated value from local logs** (itself a local-log approximation of Anthropic's window, not the window) beside the 7d %, marks the Opus cap-relative **% unavailable**, and **discloses that a near-limit alert on the 7d window may not be the window actually throttling an Opus-heavy user** — but no `opus_weekly` field or disclosure string exists in code. What the code does today: the now screen renders the **overall** 5h/7d windows (with the claude.ai chat caveat) plus the generic per-model calendar-week subscription-value rows in the cost section — Opus appears there like any other model, never with a fabricated cap-%. Only the OAuth endpoint exposes the Opus %, and that path is dropped (the single strongest reason it stays documented-but-unused, §12). Never fabricate an Opus %.
- **Costroid never reuses a stored credential, session, or token to hit a provider's undocumented or non-sanctioned endpoints** — that is the ToS line, with no exceptions. This includes Cursor, whose usage/quota are served live server-side only (no local data, no hook); with no sanctioned source, Cursor stays **detect-only** and its cost/quota are **"unavailable," never fetched** (live quota is discovery-gated — PRODUCT-PLAN.md §8). **Auth source ladder** — tiers 0–3 are the only ones ever built (preference-ordered; PRODUCT-PLAN.md §5 owns the per-provider classification): **(0) local artifacts** — logs/config, no login, no credential, no network (Claude session logs, Codex logs, Cursor presence); **(1) sanctioned push/hook** — Claude's `statusLine` `rate_limits` block, still zero network and zero credential; **(2) sanctioned OAuth** — first-party where the provider documents it (e.g. GitHub; deferred), system browser + loopback redirect with PKCE (`oauth2`), token stored to the keychain only; **(3) your own API key** — the provider's official usage/billing API (Anthropic / OpenAI / Gemini), your key, pasted by you and held in the keychain, used to reconcile the local estimate (Step 4); because such an admin key is **organization-wide**, the connect flow warns about its blast radius *before the key is pasted* (Anthropic/OpenAI only) and recommends a dedicated, instantly-revocable key; **(4) NEVER reuse any credential, session, or token against a non-sanctioned, undocumented, or internal endpoint** (this includes Cursor's `api2.cursor.sh` RPC) — that is the account-ban path and a ToS violation, so the datum stays "unavailable." **No browser-cookie reading.** Even tier (2)/(3) can't expose Anthropic *subscription* 5h/7d limits (no sanctioned API does) — the `statusLine` hook is the only route, which is why it's canonical.
- TLS via **rustls** (no OpenSSL). Network access is confined to provider endpoints the user explicitly authorized — **enforced in the type since T9a**: `costroid-connect`'s `AuthorizedClient` (the only HTTP client any Costroid build may carry, blocking `ureq` + `rustls`) is constructed over **one** authorized host and refuses any off-host URL with a typed error **before any I/O**; redirects are disabled entirely (a 3xx is a typed error, never followed), env-var proxies are ignored, requests are HTTPS-only/GET-only with bounded timeouts + body size, TLS trust comes from the **OS-native root store** (`rustls-native-certs`, never a compiled-in bundle), and caller-supplied auth headers ride in `secrecy::SecretString` so no secret can reach logs, `Debug`, or error text. The client's callers landed in T10a/T10c (the `connect`/`disconnect`/`connections` CLI and the `reconcile` display); a network call occurs **only** on an explicit, user-initiated `connect` / `connections --check` / `reconcile` action, and the default build links none of this code. **No certificate pinning:** TLS is validated against the OS-native trust store only, so a host whose trust store holds an attacker-planted or corporate-MITM-proxy root could intercept a connect/reconcile request (which carries the org-wide admin key) — the standard residual of any native-roots client; SPKI/leaf pinning for the two hardcoded hosts is a deferred hardening (disclosed in SECURITY.md). This is why the connect flow warns the key is organization-wide and recommends a dedicated, instantly-revocable key.
- **Provider logs are untrusted input** — parsed defensively; malformed data yields a clean error or "unavailable", never a crash, and the parser never executes or evaluates anything from log content.
- A **connections view** (shipped in T10a — `costroid connect`/`disconnect <provider>` + `connections [--check]`, behind `--features connect`) lists what's linked and supports instant disconnect/revoke; all network + credential code lives only in `costroid-connect`. The non-secret on-disk index (`connections.json`) records vendor slugs plus optional **org labels sanitized of control characters** at ingestion (`OrgLabel::from_server`) and is written `0600` on Unix; the secret keys themselves live solely in the OS keychain.

## 9. Non-negotiable principles (trust + don't over-engineer)

1. **The cost core is sacred.** The committed **fixture** golden tests vs. ccusage are the hard gate — if a refactor moves a number, the build fails. Real-log ccusage parity is a one-off confidence check, not a bit-identical-forever mandate (ccusage can legitimately differ).
2. **Fragile parts degrade, never crash.** Any data source that is absent, malformed, or stale returns data *or* a clean "unavailable" — never a crash. The Claude `statusLine` `rate_limits` field is **untrusted, and wrong in two distinct ways.** (a) *Out-of-range / poisoned:* a bug emits the epoch timestamp in `used_percentage` before the 5h window has data, and weekly has shown impossible values like 900% — so a value `> 100` or equal to `resets_at` means *no data*; sanitize it out. (b) *In-range but wrong:* the same field has documented false readings like a flat **100% with no actual throttling** — an in-range value no `>100` check can catch. Guard it with a **cheap divergence cross-check** (built in T4, core `finalize_limit_status`), not more sanitizing: if the statusLine % is high (≥ `0.80`) but Costroid's own local token volume for that window is trivial (< `5_000` tokens, `UNVERIFIED_TOKEN_FLOOR`), **demote it to "unverified"** rather than rendering a confident "you're maxed." The cross-check can only *flag*, not correct — a high reading may also be legitimately real (shared claude.ai-chat usage, or server-side fast-consumption) — so the local estimate is a **validator when the field is present**, not just a fallback when it's absent. **Staleness, separately, is push-only and breaks in two directions:** *high across an idle reset* is **fixable** — use `resets_at` to age a stale reading out to "unknown"; *low because quota was spent on another first-party surface* (claude.ai chat shares the same 5h/7d limit) is **not fixable, only disclosable** ("reflects only what Claude Code has observed; your true account usage may be higher"). Showing nothing or "unverified" is correct; showing a confident wrong number is fatal. The same rule governs the connect-side `reconcile` view: a hard per-vendor fetch failure (transport / oversized-or-unparseable body / keychain read — the soft 401/403/429/5xx outages already degrade inside the adapter) degrades to a **detail-free** `VendorReportUnavailable::FetchFailed` so the local estimate still renders and the *other* connected vendors still reconcile — one vendor failing never aborts the multi-vendor view, and the reason carries no string that could leak a secret or URL.
3. **Benchmarks supply quality, not cost.** They're a bundled, dated, cited snapshot — not a live fetch. DeepSWE primary (a neutral data company; scored on the neutral `mini-swe-agent` harness, *not* the user's actual Claude Code / Codex / Cursor), CursorBench corroborating (Cursor's own vendor benchmark, showcases Composer). Take the **score** axis from them; take the **dollar** axis from Costroid's own cache-correct cost — a benchmark task-average isn't the user's bill, and the published $/task vary in trustworthiness (DeepSWE's is cache-miss-priced ~5× and disputed; CursorBench's is cache-aware but vendor-computed on private tasks — §12). Where they disagree or lack a model, show both and recommend only what's supported — never invent a number. Surface source, date, and the harness/vendor caveat in the UI.
4. **Quota (%) and dollars ($) are different types.** The cross-provider total sums dollars only, per lane; quota shows alongside.
5. **Local, no content, no telemetry.** The brand and the trust both depend on it.
6. **The recommendation is a frontier view, not a prescriber.** It plots the published cost-vs-quality frontier and overlays the user's actual model mix and spend; it never claims "this task should have used X." It applies **only to API-cost rows** (model choice only changes the API bill), never subscription rows, and each recommendation carries its reasoning, its sources, and a projected dollar delta. All output is advisory.
7. **Tested against fixtures, not the network.** Unit tests per crate; integration tests driven by committed `fixtures/` logs (never real user data); `insta` snapshot tests for rendered output, especially `--plain`; and a no-network acceptance test that enforces the no-network rule. (CI gate is in §4.)
8. **Speak like a colleague.** The insight line states the fact, then the so-what, then optionally a next step — proactive, plain, specific, brief. **Hedge estimated costs** (`~`, "estimated", "about") — never false precision on inferred money. One insight at a time, quiet by default, never blocking. Sentence case; no emoji, no fake urgency, no greetings, no chatbot register.

## 10. Build sequence (let shipping be the validation)

*Throughout, **Phase 1 / Phase 2** denote data tiers — Phase 1 = local / sanctioned-push (the v1 product, built in the steps below; it already covers Claude's live 5h/7d via the `statusLine` hook and Codex's local windows, §8). The **planned connections tier** (PRODUCT-PLAN.md Step 4, the `costroid-connect` crate) adds the optional, off-by-default network sources — your own API key for reconciliation, and sanctioned OAuth where documented — each user-initiated and revocable, none on these v1 build steps. A provider with no sanctioned source (Cursor) stays "unavailable"; Costroid never reuses a session to fetch it.*

*Status (the live "where are we" lives in [PRODUCT-PLAN.md](PRODUCT-PLAN.md) §0): the v1 CLI core is built — the cost core verified to the cent vs ccusage, the frontier (`costroid frontier`) gate-green, plus WSL Windows-root auto-detect and Cursor detect-and-defer (§12). Everything beyond the v1 core is planned in PRODUCT-PLAN.md §3.*

1. **Core + workspace.** Port `costroid-core`/`-focus`/`-providers` (Claude + Codex) into the clean structure; **re-verify to the cent vs ccusage.** Fast — it's porting, not inventing. The trust foundation.
2. **TUI + full cost picture.** now/trends, subscription + API, filter, the per-lane totals, export, config (the TOML config file remains planned — zero-config today); Cursor detection (beta) and subscription quota (graceful). Ship it — **getting this adopted is the validation.**
3. **Status-line mode.** `costroid statusline` → tmux / Starship / Claude Code `statusLine`. Cheap table-stakes + a distribution channel; ship as its own small post. Call it a status-line integration, not a "cross-platform terminal toolbar" — Windows Terminal, Ghostty, and Apple Terminal have no toolbar to host. **Live Claude quota rides on this surface** (the `rate_limits` capture, §8), so a frictionless **`costroid setup-statusline`** (edits `settings.json`, handles an existing statusline) is **load-bearing, not a nice-to-have**: the flagship differentiator — subscription-quota awareness — is now gated behind this setup + Pro/Max + recent Claude Code activity, which puts onboarding friction directly on the critical path for the adoption signal.
4. **Frontier / recommendation view — built (`costroid frontier`).** The `bench` module: frontier + your-position, scoped to what the data honestly supports. Built ahead of the "added once the core has users" gate at eren's call; kept small, still cut if it doesn't land.

**Note on scope & sequencing.** The steps above are the **v1 cost-and-quota CLI core as shipped**; the going-forward, end-to-end build sequence (these steps plus the generalized quota model, the connections subsystem, the analytical tabs, and the egui taskbar) is owned by **[PRODUCT-PLAN.md](PRODUCT-PLAN.md)** §3. In that plan the **cross-platform taskbar/tray is the planned last surface** (Step 6, target v0.6.0) — built in egui/eframe + `tray-icon` (not Tauri), accessible via **AccessKit**, never relying on color alone (amber carries a second non-color cue) — the richest and most expensive surface, but one that only consumes what the core already computes.

## 11. Distribution & release

- **cargo-dist** (the installed binary is `dist`; invoke as `dist …` / `cargo dist …`); `dist init` writes the config + GitHub Actions workflow. **Nothing publishes on a normal push** — only a pushed `vX.Y.Z` tag (which must equal `[workspace.package].version`) triggers CI to build and publish; PRs run `dist plan` only. **Targets (6):** Linux gnu x86_64/aarch64, Linux musl x86_64 (static), macOS x86_64/aarch64, Windows x86_64. (cargo-dist is actively maintained — v0.32.0 — by axodotdev; if it ever stalls, fall back to hand-written installers + `cargo-binstall`, or `release-plz`.)
- Channels: shell + PowerShell installers on GitHub Releases (`releases/latest/download/costroid-installer.sh | sh`); the Homebrew tap (`Costroid/homebrew-tap`, auto-generated by cargo-dist); the npm wrapper (`npx costroid` runs the native binary, no JS runtime); crates.io — **published separately via `cargo publish` in dependency order: `costroid-focus` → `costroid-providers` → `costroid-core` → `costroid`** (see RELEASING.md), `cargo install costroid` / `cargo binstall costroid`. The `costroid-mcp` name is intentionally left unclaimed (no placeholder). **No Scoop bucket** (cargo-dist doesn't support it) — Windows users use the PowerShell installer or `cargo binstall`; a hand-maintained bucket is a later, only-on-demand item.
- **Signing:** releases are **not OS-code-signed**; each artifact ships **keyless GitHub build-provenance attestations + SHA-256 checksums** instead (verify with `gh attestation verify <file> --repo Costroid/costroid`). The trade-off: first run shows a macOS "unidentified developer" / Windows SmartScreen prompt. To enable signing later (paid, and just config toggles on the existing pipeline): macOS notarization (Apple Developer ID, ~$99/yr) + Windows Authenticode (EV cert with HSM/cloud signing, ~$200–300/yr), then cargo-dist's signing config + the matching secrets. Revisit only if those prompts become a real adoption problem.

## 12. The honest flags, in one place

- Don't literally rewrite from zero — **port** the validated cost core; it's the trust asset and the bug-trap you already escaped.
- The tray/taskbar is the **planned last surface** (egui/eframe + `tray-icon`, not Tauri; PRODUCT-PLAN.md Step 6), after the CLI core; note "cross-platform terminal toolbar" still means the status-line integration, *not* the taskbar (most terminals have no toolbar to host).
- Pricing lives inside `costroid-core`, never a top-level directory — cargo won't package it otherwise.
- Your live-quota fragility risk is the **Claude `statusLine` `rate_limits` field** (absent, poisoned, *and* false-in-range — needs sanitize **+** local cross-check, §9.2); it must degrade to "unavailable"/"unverified," never a confident wrong number. Cursor has no live source at all (no session reuse — §8), so its quota is simply "unavailable."
- Recommendation = frontier + your position, not per-task prescriptions; DeepSWE over CursorBench on neutrality.
- The "bigger picture total" is per-lane (API $ / subscription-equivalent $ / quota %), never one merged number.
- Sparkline is hand-rasterized (no Ratatui `Canvas`) — one styling path feeds both the TUI and `--plain`; `NO_COLOR` drops ANSI styling only (braille fill stays glyph-distinct — never color alone; ASCII is the braille-incapable fallback). Both **settled** (mechanics in §7).
- Iterate the existing crates (published through 0.3.0; the 0.4.0 ladder adds `costroid-connect` — T10b); don't orphan the claimed names.
- **Opus real-log quirk:** costroid runs ~0.08% under ccusage on opus totals — isolated to re-logged sub-agent (sidechain) cache-read de-dup; mainline matches to the cent. Benign methodology difference, Claude parser unchanged (§9.1); the invoice is ground truth (Phase 2+).
- **WSL Windows-root discovery (fixed):** under WSL with `USERPROFILE` unset, costroid scans `/mnt/c/Users/*` for profiles holding `.claude`/`.config/claude`/`.codex`/`.cursor` and merges them — and Codex discovery now merges all roots (was first-root-wins), with session-level cross-root dedup. Residual behaviors to know: a *set* `USERPROFILE` (even empty) is explicit → no scan; the scan is `/mnt/c` only (other drives need the env knobs); an empty scan falls back to the legacy `/mnt/c/Users/$USER` guess; it's evidence-based, so it includes *any* Windows profile with logs (e.g. a sandbox profile).
- **Claude live quota is sanctioned, not reverse-engineered (verified 2026-06-05).** Source = Claude Code's documented `statusLine` hook (the `rate_limits` block: `five_hour`/`seven_day` → `used_percentage` + `resets_at`; Pro/Max, after the first API response; zero API tokens), captured via the paste-in snippet or by being the statusline (§8) — never by reusing Claude Code's token. So Costroid shows the **real** 5h/7d numbers with no account/ToS risk — the opposite of codexbar. The field is **buggy** (§9.2): **absent** for API keys; **poisoned** (epoch in `used_percentage`); shows **impossible (900%)** and **false-in-range (a flat 100% with no throttling)** values — so Costroid **sanitizes** (`>100` / `==resets_at`, at parse time) *and* **cross-checks** a high reading against local volume, demoting it to "unverified" rather than trusting it (both built in T4, §9.2). It's **push-only** (label "as of HH:MM / stale"; `refreshInterval` re-renders but does **not** freshen quota — the field only moves on a new API response), reads **low** if quota was spent on claude.ai chat (disclose, not fixable), and omits the **per-model weekly windows** — so lead with the overall 7d % as the measurable number (not the binding cap; for a ~97%-Opus user the Opus weekly may bind first); the companion Opus-specific render — Opus 7d volume/value (a local-log approximation) with the cap-% **unavailable**, plus the 7d-window-may-not-be-the-throttling-window disclosure — is **design intent, not yet shipped** (§8). Parse `resets_at` defensively (seen as both epoch seconds and ISO across versions). This **supersedes** the `/api/oauth/usage` path for Claude; that path is dropped entirely, and there is no session-reuse plan for any provider (Cursor included — its live quota is discovery-gated, §8).
- **Cursor is detection-only, and stays that way until a sanctioned source exists** (no local data, no sanctioned hook; its usage/quota are served live server-side, and Costroid never reuses a session to fetch them). Phase 1 shows Cursor present + selected model, **beta**, usage/quota *"unavailable — no sanctioned source."* Live Cursor quota is **discovery-gated (PRODUCT-PLAN.md §8)** — pursued only if Cursor publishes a documented API or first-party OAuth, never via session reuse against `api2.cursor.sh`. The quota *shape* is already resolved and supported: Cursor's paid plans are a **monthly, dollar-denominated credit pool + usage-based overage** (a `BillingCycle`, spend-$ window), and the **daily token window is the free-tier rate-limit** (not "daily, spend-$"); the generalized model (§6, PRODUCT-PLAN.md §2a) supplies the `Monthly`/`BillingCycle` kinds and the `Spend` measure that fit it (landed in T2). Only a sanctioned *source* is missing. Cursor spend is a *spend* window, never summed into the cost lanes.
- **The two benchmarks have different weaknesses (verified 2026-06-04).** DeepSWE (updated 2026-05-30) is a **neutral** source, but its avg cost/task prices input at the cache-*miss* rate — a ~5× overstatement vs cache-correct pricing — and scores on the neutral `mini-swe-agent` harness, *not* Claude Code/Codex/Cursor (the harness swings scores ~±10 pts). CursorBench v3.1 (2026-05-18, `cursor.com/cursorbench`) is the opposite: a **vendor** benchmark (Cursor's own private, non-reproducible tasks, showcases Composer), but its cost is **cache-aware** — computed from each model's published per-token pricing including cache reads — so its $/task is methodologically sound; the caveat there is provenance, not cost. So step 4 takes the **quality/score** axis from both (DeepSWE primary, CursorBench corroborating) and the **dollar** axis from Costroid's own cache-correct cost (a task-average isn't the user's bill); un-benchmarked models (e.g. opus-4.8) show as gaps, never guessed (§9.3). Verified figures (all exact): DeepSWE avg cost/task (cache-miss-priced) gpt-5.5 70% @ $6.61, opus-4.7 54% @ $18.19, sonnet-4.6 32% @ $5.52; CursorBench (cache-aware) composer-2.5 63.2% @ $0.55, opus-4.7 Max 64.8% @ $11.02, gpt-5.5 64.3% @ $4.37.
